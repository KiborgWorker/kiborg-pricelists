## About

**Kiborg Pricelists** - сервіс, призначений для автоматизації формування прайс-листів для дропшиперів. Він збирає дані про товари з різних джерел, об’єднує їх і самостійно формує зручні списки для передачі клієнту.

Цей сервіс було спроектовано та розроблено для заміни старої системи, в якій додавання та контроль товарів здійснювалися вручну. Головними завданнями були:
- автоматизація роботи з товарами;
- оптимізація клієнтської частини;
- поділ клієнта за компонентною архітектурою для зручної підтримки в майбутньому.
<br>

## Get Started

### Local Start
Щоб запустити проект локально, спочатку потрібно скопіювати репозиторій за допомогою команди:
``` bash
git clone https://github.com/KiborgWorker/kiborg-pricelists.git
```

Далі відкриваємо папку проекту в терміналі.

#### Робота з сервером
Щоб запустити локальний сервер, потрібно:
1. Перейти до папки з сервером
```
cd ./server/
```
2. Завантажити всі необхідні для сервісу пакети
```
npm install
```
3. Запустити сервер
```
npm run dev
```
У консолі з’явиться посилання, яке використовується для роботи з API.
<br>

#### Робота з клієнтами
У проєкті є три клієнти:
```
- price-all     - уся продукція;
- price-kiborg  - продукція бренду Kiborg;
- price-militex - продукція бренду Militex.
```
<br>

Щоб запустити потрібний клієнт локально, потрібно:
1. Перейти до відповідної папки цього клієнта
``` bash
cd ./client/price-all
```
або
``` bash
cd ./client/price-kiborg
```
або
``` bash
cd ./client/price-militex
```
<br>

2. Встановити всі необхідні пакети:
``` bash
npm install
```

2. Запустити dev-сервер
``` bash
npm run dev
```

Після чого в консолі з’явиться посилання на сайт, де всі зміни будуть одразу застосовані.

> [!note] Примітка
> За замовчуванням клієнти підключаються до вже розгорнутого сервера. Щоб використовувати локальний сервер, потрібно змінити у файлі `./src/api/api.ts` наступний рядок:
> ``` ts
> const baseUrl = “SERVER_URL”;
> ``` 
> У ній необхідно вставити посилання на сервер, і <font color="#c00000">ВАЖЛИВО</font> щоб посилання завантажувалося з `/`
<br>

### Deployment
Після виконання будь-яких робіт локально, їх потрібно завантажити на GitHub. **НЕ РЕКОМЕНДУЄТЬСЯ** завантажувати зміни одразу у гілку `master`, оскільки вона автоматично оновлює клієнтські частини, і якщо там було щось не так, це одразу відображається у результуючому продукті. Краще створити окрему гілку, а потім скористатися `pull request`.
<br>

Усі клієнти розміщені на **Vercel**, і до них прив’язані зовнішні домени. Як уже згадувалося раніше, усі зміни клієнтів у гілці `master` автоматично застосовуватимуться до готового продукту. Тому подальших налаштувань не потрібно.
<br>

Сервер, у свою чергу, розміщений на **Render**, тому має деякі тонкощі в налаштуванні. Він також прив'язаний до гілки `master`, але тут після змін у серверній частині репозиторію потрібно робити ручний ре-деплой. 
<br>

Для цього потрібно зайти на сторінку сервера на Render. Відкрити вкладку **Events**. У правому верхньому куті є кнопка **Manual Deploy**, після натискання якої з'являється меню, в якому потрібно вибрати `Clear build cache & deploy`. Після цього підтягнуться всі оновлення, що були зроблені раніше.
<br>

> ВАЖЛИВО!
> Через особливості платформи та архітектури проєкту під час кожного компілювання проєкту база даних стирається, тому для повноцінного функціонування потрібно примусово оновити базу даних. Для цього перейдіть за посиланням
> ```
> https://kiborg-server-for-products-pricelists.onrender.com/update
> ```



## Architecture

Ця система складається з таких основних компонентів:
- джерела даних;
- сервер;
- база даних;
- клієнтська частина:
    - прайс-лист для всієї продукції на дроп;
    - прайс-лист для продукції бренду Kiborg:
    - прайс-лист для продукції бренду Militex.
<br>

### Джерела даних
Джерела являють собою вивантаження з різних сервісів у форматі XML. Кожне містить певну інформацію про товари. Всього є, по суті, 6 джерел

- Вивантаження з **Tactic Shop** — вся інформація про товари (артикул, назва, опис, категорія тощо);
- Вивантаження з **Sales Drive** — роздрібні ціни;
- Вивантаження з **Sales Drive** — оптові ціни;
- Вивантаження з **Sales Drive** — дроп-ціни;
- Вивантаження з **Kiborg** - посилання на товари на сайті Kiborg;
- Вивантаження з **Militex** - посилання на товари на сайті Militex.
<br>

### Сервер
Сервер отримує дані з джерел, аналізує, обробляє, фільтрує та зберігає їх у базі даних SQLite. Далі він надає всі необхідні дані клієнтам через API.
<br>

Сервер було реалізовано з використанням таких технологій:
- **Node.js**
- **Express.js**
- **TypeScript**
<br>

Основні компоненти сервера:
- Парсер — перетворює отримані XML-файли на той тип даних, з яким йому вже зручно працювати
- Обробка даних — під час обробки відбувається агрегація даних з усіх джерел і подальша фільтрація відповідно до вимог
- Функціонал для роботи з БД — інтерфейс для зручного запису інформації в БД та її отримання при запитах від клієнта 
- API — функціонал для взаємодії клієнта з сервером.
<br>

Також для забезпечення автоматичного оновлення даних про товари було налаштовано всі необхідні функції за допомогою CRON-завдань.
<br>

### База даних
База даних реалізована за допомогою SQLite і складається з двох таблиць: Products, Categories. Також до цього пункту можна віднести JSON-файл з конфігурацією видимих категорій.

### Клієнтські частини
Всі три клієнтські частини мають практично ідентичну структуру. Різниця лише в параметрах запитів, а також у тому, що для бренду Militex має дещо іншу стилістику.

Клієнт був розроблений за допомогою
- **React**
- **TypeScript**
- **TailwindCSS**
- **Shadcn**

Завдяки чому була побудована досить проста архітектура та зручний розподіл за компонентами.
<br>

## Project Structure

Ця система складається з 4 сервісів, розподілених за такою структурою:
```
server/           — сервіс бекенду (API + обробка даних)
  
client/
  price-all/      - Прайс-лист усієї продукції
  price-kiborg/   - Прайс-лист бренду Kiborg
  price-militex/  - Прайс-лист бренду Militex  
```
<br>

`server/` у свою чергу має такий вигляд:
```
package.json
other_additional_files...
src/
    index.ts    - точка входу сервера (ініціалізація та запуск)
    database/   - робота з базою даних (ініціалізація, запити)  
    parsers/    - парсинг XML-джерел  
    services/   - бізнес-логіка та агрегація даних  
    store/      - конфігураційні файли  
    types/      - типи TypeScript  
    utils/      - допоміжні функції
```
<br>

Усі клієнтські додатки мають ідентичну структуру:
```
package.json
index.html
other_additional_files...
public/
src/
    api/        - логіка для роботи з API сервера
    assets/     - зображення, іконки, шрифти
    components/ - компоненти, що повторно використовуються
    lib/        - додаткова логіка від Shadcn
    styles/     - стилі
    types/      - типи даних
    App.tsx     - головний файл для відображення сторінки клієнта
    main.tsx    - підключення React
```
<br>

## Data Flow

1. Сервер отримує XML-дані з 6 джерел
2. Аналізує та об’єднує їх у єдину структуру
3. Зберігає в SQLite
4. Клієнти отримують дані через API
<br>

## API

Нижче описано основні API-запити, реалізовані на сервері
<br>

### 🔄 GET `/update`
Запускає оновлення даних (парсинг джерел і запис у БД). Автоматично виконується раз на добу за допомогою CRON-завдання.
<br>

⚠️ Використовується після деплою або при необхідності оновлення прайс-листів.
<br>

### 📊 GET `/status`
- Перевірка працездатності сервера.
- Використовується для підтримки працездатності сервера за допомогою CRON-завдання.
<br>

### 📂 GET `/get/categories`
Повертає список категорій з інформацією про наявність товарів.
#### Параметри запиту:
- `vendorFilter` _(string, optional)_ — фільтр за брендом (наприклад: `Kiborg`, `Militex`)
#### Приклад запиту:
```
GET /get/categories?vendorFilter=Kiborg
```
#### Відповідь:
``` json
[
  {
    «id»: 1,
    «name»: «Category name»,
    «isEmpty»: false
  }
]
```
<br>

### 📦 GET `/get/products`
Отримання товарів.
⚠️ Обов'язковий один із параметрів:
- `productId`
- `categoryId`

#### 1. Отримання товару за ID
```
GET /get/products?productId=123
```
##### Відповідь
``` json
{
  «id»: 123,
  «name»: «Назва товару»,
  «price»: 100
}
```

#### 2. Отримання товарів за категорією
```
GET /get/products?categoryId=1
```

##### Параметри запиту:
- `categoryId` _(number, required)_ — ID категорії
- `vendorFilter` _(string, optional)_ — фільтр за брендом

##### Приклад:
```
GET /get/products?categoryId=1&vendorFilter=Militex
```

##### Response:
``` json
[
  {
    "id": 1,
    "name": "Product",
    "price": 100,
    "url": "https://example.com"
  }
]
```


> Важливо
>
> - Параметри `productId` та `categoryId` є взаємовиключними 
> - Якщо не передано жодного параметра — повернеться помилка 400 
> - Поле `urlMilitex` приховується у більшості відповідей 
> - Відповіді можуть бути відсортовані у зворотному порядку


## CRON

Усі заплановані завдання розміщені на сервісі `https://console.cron-job.org/`.
Всього є 4 основні завдання
- Пробудження парсингу сервера — виконує запит на сервер кожні 5 хвилин, щоб той не заснув.
- Оновлення списку товарів — запускає оновлення бази даних раз на добу, вночі.
- Генерація вивантаження з Kiborg - запускає оновлення вивантаження з сайту Kiborg. Також запускається раз на добу вночі за деякий час до оновлення бази даних
- Генерація вивантаження з Militex - аналогічна до попередньої, тільки з сайту Militex.
<br>

## Common Issues

### Не завантажуються прайс-листи (замість категорій відображається нескінченний Loader).
Найімовірніше, сервер недоступний. Оскільки сервер розміщений на безкоштовній платформі Render, іноді може траплятися так, що CRON-завдання з підтримки працездатності повертає помилку, після чого припиняє свою роботу. Оскільки на сервер не надходять запити, він «засинає»; щоб він «прокинувся», потрібно зробити будь-який запит і почекати деякий час. Після цього також потрібно перезапустити CRON-завдання.

Також таке може статися, якщо під час розгортання виникли якісь помилки, і тому сервер не запускається. Тоді вже потрібно перевірити, що за помилка, виправити її або повернутися до попередньої версії та перезапустити сервер.

### Усі категорії порожні
Швидше за все, база даних порожня. Потрібно просто примусово її оновити. Найчастіше таке трапляється після ре-деплою, але іноді така проблема може бути пов'язана з попередньою.
